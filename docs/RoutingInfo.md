## DAPX Architecture / **Routing information**



This document describes the structure and distribution of routing information in the [DAPX](../README.md) protocol. Routing information tells a sender how to reach a recipient through the available [delivery paths](./MessageFlow.md).



# Routing information



Routing information is generated by the recipient and shared with others. It contains everything a sender needs to deliver messages without knowing the recipient's network address directly.



---



## Contents



A recipient's full routing information contains:

- a **friendship token**, unique to this specific relationship (see [friendship tokens](#friendship-tokens) below)

- the addresses of the recipient's **mailbox nodes**

- the addresses of the recipient's **relay nodes**

- the recipient's **segmentation configuration** (so the sender knows how to split messages across mailbox nodes)

- the recipient's **preferred message TTL** for mailbox storage

- a **version timestamp** used to determine ordering when routing information is updated



---



## Two-tier model



Routing information exists in two forms.


### Private routing information

This is the full set of routing information as described above. It is shared only with established friends during [friendship establishment](./FriendRequest.md). Every friend receives the same private routing information.


### Public routing hint

This is a minimal subset published to [index nodes](./DiscoveryFlow.md) for users who have opted into discovery. It contains just enough for a stranger to deliver a [friend request](./FriendRequest.md): one or more mailbox or relay addresses. It does not include a friendship token, segmentation details, or the full node list.

No friendship token is needed for friend request delivery. Both [mailbox nodes](./MailboxProtocol.md) and [relay nodes](./RelayProtocol.md) accept friend requests for users they serve without requiring a token, as long as the user has permitted friend request delivery.

A user who has not opted into discovery does not publish any public routing hint. In that case, friend requests can only be delivered through a direct connection.



---



## Friendship tokens



A friendship token is an opaque value that authorizes a specific sender to use the recipient's infrastructure nodes. Each friendship produces a unique token. If A and B become friends, A receives a token from B and B receives a token from A. These tokens are different from each other and different from tokens issued to any other friend.

The sender includes the friendship token in the outer envelope of every packet (see [Packet Format](./PacketFormat.md)). Infrastructure nodes, such as [mailbox nodes](./MailboxProtocol.md) and [relay nodes](./RelayProtocol.md), check the token against a list of valid tokens for the user they serve. If the token is valid, the node processes the packet. If not, the node rejects it.

Because tokens are unique per relationship, a user can revoke one friend's access by invalidating their token without affecting any other friend's ability to reach them. Token revocation happens during [unfriending](./FriendRequest.md).



---



## Updates



When a user changes their routing information (for example, switching mailbox nodes or changing segmentation), they send the updated routing information to their friends. This is delivered as a standard [update](./PacketFormat.md) packet through whatever delivery path is available. If a friend is offline, the update is deposited on their mailbox.

The version timestamp in the routing information allows the receiving client to determine whether an update is newer than what it already has.

Routing information updates to friends are separate from [configuration updates](./Configuration.md) sent to infrastructure nodes. Both may result from the same change (for example, adding a new mailbox node), but they travel through different channels: config updates go to nodes, routing info updates go to friends.
